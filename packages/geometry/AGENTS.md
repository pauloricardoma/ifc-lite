# Agent notes: packages/geometry

Supplements the root [AGENTS.md](../../AGENTS.md) Geometry & WASM section. This package is the TS wrapper around the wasm kernel: it copies WASM getters into JS arrays and hands off for GPU upload. Package-specific footguns:

- **Domain logic is Rust, at the FFI boundary.** Decode, meshing, color, and the IFC Z-up to renderer Y-up axis swap all happen in Rust (`rust/processing/src/element.rs`); meshes arrive here already in Y-up. Do not add axis flips or decode logic in this package.
- **Free every WASM handle deterministically.** Extract each mesh, then `mesh.free()` in an inner `finally` and `collection.free()` in an outer `finally`; release the pre-pass cache via `api.clearPrePassCache?.()` in `finally` so it runs on a throw, the `totalJobs === 0` early return, or generator abandonment (`.return()`). Getters copy WASM memory into JS arrays, so freeing right after extraction is safe (`src/geometry-coordinate.ts`, `src/index.ts:444`).
- **Mesh extraction is duplicated on two paths that must stay in sync:** `convertMeshCollectionToBatch` (main thread, `collection.get(i)` copy) and the worker's `collectMeshes` (`collection.takeMesh(i)` move, one-read). Any new `MeshData` field (shadingColor, origin, localBounds, localToWorld, geometryClass, texture) must be added to BOTH or the default worker load path silently drops it.
- **Keep the Tauri path web-pure.** `@tauri-apps/api` is an optionalDependency; its only import is a lazy `await import('@tauri-apps/api/event')` inside `NativeBridge`, and `createPlatformBridge()` is reached exclusively under `isTauri()`. Never add a top-level `@tauri-apps` import.
- **WASM streaming is single-flight and fatal errors are terminal.** A module-global `activeWasmStreamingOperation` mutex makes a second `processStreaming` throw while one runs (shared pre-pass cache). After a fatal WASM runtime error, `init()` re-throws forever, so recover by reloading the page or recreating the worker, not in-process.
- **Worker count is memory-budgeted, not core-count.** `computeWorkerCount` (`src/worker-count.ts`) uses `SMALL_FILE_MB=24` / `MIN_PARALLEL_MB=8` bands and clamps every path (including the `?geomWorkers` override) by `memoryCap`. Each worker is a separate WASM instance that re-decodes the whole file into its own heap (~5x peak memory), so do not widen the fast-path band without a worker-count sweep.
- **`batch-sizing.ts` and `watchdog.ts` are coupled.** They are pure-function extractions of worker logic (the worker imports WASM at top level and isn't unit-testable): `DEFAULT_BATCH_SIZING` (targetMs 8000 / min 64 / max 512) and `SUBSEQUENT_BATCH_MS_BROWSER` (40000). `maxJobs * worst-ms-per-job` must stay under the subsequent-batch watchdog grace, so changing one requires re-checking the other.
- **The worker is instantiated by `.ts` path** via `new Worker(new URL('./geometry.worker.ts', import.meta.url))`; the build script string-replaces `geometry.worker.ts` to `.js` in the dist output. Bundlers (Vite/webpack5) rewrite the wasm-bindgen URL literal automatically; other consumers must pass `wasmUrls.wasm`.
- **The real boundary test is repo-root `pnpm test:wasm-contract`** (loads the actual `packages/wasm/pkg`). In-package vitest mocks `@ifc-lite/wasm`, so green package tests prove nothing about the WASM contract. `test:wasm-contract` itself skips clean (exit 0) if the wasm runtime isn't built, so run `scripts/build-wasm.sh` first locally or that green proves nothing either.
